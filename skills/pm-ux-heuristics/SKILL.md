# UX 휴리스틱 & 법칙

## 닐슨 10가지 휴리스틱

기획 단계와 개발 중 의사결정 시 아래 10가지 원칙을 체크리스트로 사용한다.

### 1. 시스템 상태 가시성 (Visibility of System Status)

시스템은 현재 무슨 일이 일어나고 있는지 사용자에게 항상 알려야 한다.

**체크리스트:**
- [ ] 데이터 로딩 중임을 표시하는가? (스피너, 스켈레톤, 프로그레스바)
- [ ] 작업 완료를 알려주는가? (성공 토스트, 체크 아이콘)
- [ ] 진행률을 보여주는가? (파일 업로드, 멀티스텝 폼)
- [ ] 현재 위치를 알 수 있는가? (브레드크럼, 활성 메뉴)

**위반 예시:**
- 버튼 클릭 후 아무 반응 없이 3초 대기
- 폼 제출 후 성공/실패 피드백 없음
- 파일 업로드 중 진행률 표시 없음

### 2. 현실 세계와 일치 (Match Between System and Real World)

시스템 용어가 아닌 사용자가 익숙한 언어와 개념을 사용한다.

**체크리스트:**
- [ ] 기술 용어 대신 사용자 친숙한 용어를 쓰는가?
- [ ] 아이콘이 현실 세계의 관례를 따르는가?
- [ ] 정보가 자연스럽고 논리적인 순서로 나열되는가?

**위반 예시:**
- "401 Unauthorized" 대신 "로그인이 필요합니다"를 사용해야 함
- "Query parameter invalid" 같은 개발자 에러 메시지 노출
- 날짜 형식이 지역 관례와 다름 (MM/DD/YYYY → YYYY.MM.DD)

### 3. 사용자 통제와 자유 (User Control and Freedom)

사용자가 실수로 원치 않는 상태에 빠졌을 때 쉽게 빠져나올 수 있어야 한다.

**체크리스트:**
- [ ] 실행 취소(Undo) 기능이 있는가?
- [ ] 취소(Cancel) 버튼이 명확히 보이는가?
- [ ] 모달/대화상자를 쉽게 닫을 수 있는가? (ESC, 배경 클릭)
- [ ] 뒤로가기가 정상 동작하는가?
- [ ] 작업 중단 시 데이터가 보존되는가?

**위반 예시:**
- 삭제 후 복구 불가
- 모달에 닫기 버튼 없음
- 뒤로가기 시 입력 데이터 소실

### 4. 일관성과 표준 (Consistency and Standards)

동일한 상황에서 동일한 단어, 아이콘, 동작을 사용한다.

**체크리스트:**
- [ ] 같은 기능에 같은 용어를 쓰는가? ("삭제" vs "제거" 혼용 금지)
- [ ] 버튼 스타일이 일관적인가? (Primary/Secondary 구분)
- [ ] 같은 유형의 동작이 같은 위치에 있는가?
- [ ] 플랫폼 관례를 따르는가? (웹 표준, OS 관례)

**위반 예시:**
- "저장" 버튼이 어떤 페이지는 우측 상단, 다른 페이지는 하단
- 같은 기능을 "수정", "편집", "변경"으로 혼용
- 확인 버튼이 어떤 곳은 좌측, 어떤 곳은 우측

### 5. 에러 예방 (Error Prevention)

에러 메시지보다 에러가 발생하지 않도록 설계하는 것이 우선이다.

**체크리스트:**
- [ ] 위험한 동작 전 확인 대화상자가 있는가? (삭제, 결제)
- [ ] 입력 제약 조건을 사전에 보여주는가? (비밀번호 규칙 등)
- [ ] 불가능한 옵션은 비활성화 되어 있는가?
- [ ] 실시간 유효성 검사를 제공하는가?
- [ ] 자동 저장 기능이 있는가?

**위반 예시:**
- "정말 삭제하시겠습니까?" 확인 없이 즉시 삭제
- 비밀번호 입력 후에야 "8자 이상" 규칙을 알려줌
- 재고 없는 상품의 "구매" 버튼이 활성화 되어 있음

### 6. 기억보다 인지 (Recognition Rather Than Recall)

사용자가 기억해야 할 것을 최소화하고, 옵션과 정보를 눈에 보이게 한다.

**체크리스트:**
- [ ] 자주 쓰는 옵션이 바로 보이는가?
- [ ] 최근 검색/사용 기록을 제공하는가?
- [ ] 선택지를 목록으로 보여주는가? (자유 입력보다 선택)
- [ ] 도움말/힌트가 필요한 곳에 있는가?

**위반 예시:**
- 검색 기록 없이 매번 처음부터 입력
- 코드값을 직접 입력해야 함 (드롭다운 대신)
- 이전 단계의 선택 내용이 안 보임

### 7. 유연성과 효율 (Flexibility and Efficiency of Use)

초보자와 숙련자 모두를 위한 경로를 제공한다.

**체크리스트:**
- [ ] 키보드 단축키를 지원하는가?
- [ ] 자주 쓰는 기능에 빠른 접근이 가능한가?
- [ ] 고급 사용자를 위한 일괄 처리 기능이 있는가?
- [ ] 개인화/커스터마이즈가 가능한가?

**위반 예시:**
- 100개 항목을 하나씩만 삭제 가능 (일괄 삭제 없음)
- 자주 쓰는 기능이 3단계 메뉴 깊이에 숨겨져 있음

### 8. 심미적/미니멀 디자인 (Aesthetic and Minimalist Design)

불필요한 정보를 제거하고 핵심에 집중한다.

**체크리스트:**
- [ ] 각 화면에서 핵심 작업이 명확한가?
- [ ] 불필요한 장식 요소가 없는가?
- [ ] 정보의 우선순위가 시각적으로 구분되는가?
- [ ] 한 화면에 너무 많은 기능이 있지 않은가?

**위반 예시:**
- 대시보드에 20개 이상의 카드
- 모든 텍스트가 같은 크기와 굵기
- 사용하지 않는 메뉴가 항상 표시

### 9. 에러 인식/진단/복구 (Help Users Recognize, Diagnose, and Recover from Errors)

에러 메시지는 문제를 명확히 설명하고 해결 방법을 제시해야 한다.

**체크리스트:**
- [ ] 에러 메시지가 기술 용어 없이 이해 가능한가?
- [ ] 에러의 원인을 구체적으로 설명하는가?
- [ ] 해결 방법을 제시하는가?
- [ ] 에러 발생 위치 근처에 메시지가 표시되는가?

**위반 예시:**
- "Error 500" (원인/해결 방법 없음)
- "입력값이 유효하지 않습니다" (어떤 입력인지 불명확)
- 페이지 상단에만 에러 표시 (긴 폼에서 스크롤 필요)

### 10. 도움말과 문서 (Help and Documentation)

도움 없이 사용할 수 있는 것이 최선이지만, 필요한 곳에 도움말을 제공한다.

**체크리스트:**
- [ ] 복잡한 기능에 툴팁/도움말이 있는가?
- [ ] 도움말이 검색 가능한가?
- [ ] 작업 중심으로 구성되어 있는가? (기능 나열이 아닌)
- [ ] 온보딩/튜토리얼이 있는가? (첫 사용자용)

**위반 예시:**
- 복잡한 설정 화면에 설명 없음
- FAQ만 있고 검색 기능 없음

## UX 법칙

기획 의사결정 시 참조하는 핵심 UX 법칙:

### Fitts's Law (피츠의 법칙)

> 타겟까지의 거리가 가까울수록, 타겟이 클수록 빠르게 도달한다.

**적용:**
- 자주 쓰는 버튼(CTA)은 크게, 접근하기 쉬운 위치에 배치
- 관련 있는 요소는 가까이 배치 (폼 필드와 에러 메시지)
- 모바일에서 터치 타겟 최소 44x44px

### Hick's Law (힉의 법칙)

> 선택지가 많을수록 결정에 걸리는 시간이 증가한다.

**적용:**
- 한 화면에 선택지 최소화 (5-7개 이하)
- 카테고리로 그룹화하여 단계별 선택 유도
- 기본값 제공으로 선택 부담 감소
- 검색/필터 제공으로 대량 옵션 관리

### Jakob's Law (야콥의 법칙)

> 사용자는 대부분의 시간을 다른 사이트에서 보낸다. 익숙한 패턴을 따르면 학습 비용이 줄어든다.

**적용:**
- 업계 표준 UI 패턴 사용 (장바구니 아이콘 = 우측 상단 등)
- 일반적인 내비게이션 구조 유지
- 혁신보다 관례 우선 (창의적이되 혼란을 주지 않게)

### Miller's Law (밀러의 법칙)

> 사람은 단기 기억에 7(±2)개의 항목을 보유할 수 있다.

**적용:**
- 정보를 5-9개 단위로 청킹(chunking)
- 긴 목록은 그룹으로 나누기
- 멀티스텝 프로세스에서 현재 단계 표시
- 한 번에 보여주는 정보량 제한

## 기획 단계 적용 가이드

기능을 기획할 때 아래 순서로 휴리스틱을 점검한다:

### 필수 체크 (모든 기능)
1. **시스템 상태 가시성** — 로딩/성공/에러 상태 정의했는가?
2. **에러 예방** — 사용자 실수를 방지하는 장치가 있는가?
3. **에러 복구** — 에러 메시지가 해결 방법을 포함하는가?
4. **일관성** — 기존 화면/패턴과 일관적인가?

### 권장 체크 (복잡한 기능)
5. **사용자 통제** — 실행 취소/뒤로가기가 가능한가?
6. **기억보다 인지** — 사용자가 기억해야 할 것을 최소화했는가?
7. **유연성** — 다양한 사용자 수준을 고려했는가?

### 선택 체크 (고도화 시)
8. **미니멀 디자인** — 불필요한 정보를 제거했는가?
9. **현실 일치** — 사용자 친숙한 용어/개념인가?
10. **도움말** — 복잡한 기능에 가이드가 있는가?
