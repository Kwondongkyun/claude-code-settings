# 기능 명세 & 요구사항

## PRD (Product Requirements Document) 템플릿

모든 기능 기획은 아래 구조를 따른다. 규모에 따라 섹션을 생략할 수 있으나, 필수 항목은 반드시 포함한다.

### PRD 구조

```markdown
# [기능명] PRD

## 1. 개요 (필수)
- **목적**: 이 기능이 해결하는 문제
- **배경**: 왜 지금 이 기능이 필요한가
- **대상 사용자**: 누가 사용하는가
- **성공 지표**: 어떻게 성공을 측정하는가

## 2. 기능 요구사항 (필수)
### 2.1 핵심 기능
- FR-001: [기능 설명]
  - 수용 기준: Given/When/Then
- FR-002: [기능 설명]
  - 수용 기준: Given/When/Then

### 2.2 부가 기능
- FR-101: [기능 설명]

## 3. 비기능 요구사항
- 성능: 응답 시간 N초 이내
- 보안: 인증/인가 요구사항
- 접근성: WCAG 2.2 AA 준수
- 호환성: 지원 브라우저/디바이스

## 4. 화면 목록
| # | 화면명 | 설명 | 진입 경로 |
|---|--------|------|----------|
| 1 | ... | ... | ... |

## 5. 상태 정의 (필수)
상태를 정의하지 않으면 개발 중 빈 화면, 무한 스피너, 에러 미처리 같은 버그가 남는다.
각 화면별 5가지 상태(Default/Loading/Empty/Error/Success)를 정의한다.
상세 규칙(로딩 UI 선택, Empty CTA, 에러 메시지 원칙 등)은 `pm-error-scenarios` 스킬이 정의의 주인이다. 여기서는 상태 목록만 나열.

## 6. 우선순위
MoSCoW 기준으로 분류

## 7. 릴리즈 계획
- Phase 1 (MVP): ...
- Phase 2: ...
```

## 수용 기준 작성 가이드

### Given-When-Then 포맷

모든 기능 요구사항에 수용 기준을 반드시 포함한다.

```
Given: [사전 조건 — 시스템/사용자의 현재 상태]
When: [트리거 — 사용자가 수행하는 행동]
Then: [결과 — 시스템이 보여주는 응답]
```

### 작성 규칙

1. **하나의 수용 기준 = 하나의 시나리오** (여러 시나리오를 하나에 합치지 않는다)
2. **테스트 가능해야 한다** (주관적 표현 금지: "빠르게", "적절하게" → 구체적 수치)
3. **사용자 관점으로 작성** (기술 구현이 아닌 사용자가 보는 결과)
4. **에러 메시지 문구는 직접 정의하지 않는다** — 에러 메시지 작성 원칙(명확성, 해결 방법, 기술 용어 금지)은 `pm-error-scenarios`가 정의의 주인. 수용 기준에서는 "에러 메시지가 표시된다"까지만 명시

### 수용 기준 체크리스트

- [ ] 정상 동작 (Happy Path)이 정의되어 있는가?
- [ ] 실패 케이스가 정의되어 있는가?
- [ ] 경계값이 정의되어 있는가? (최소/최대)
- [ ] 권한별 동작이 정의되어 있는가?
- [ ] 응답 시간/성능 기준이 있는가?

## 우선순위: RICE + MoSCoW

### RICE 점수 산출

각 FR(기능 요구사항)에 RICE 점수를 부여하여 구현 순서를 결정한다.

| 요소 | 설명 | 범위 |
|------|------|------|
| **R** (Reach) | 이 기능이 영향을 미치는 사용자 규모 | 1~10 (1=소수, 5=절반, 10=전체) |
| **I** (Impact) | 개별 사용자에 대한 가치 크기 | 0.25(최소) / 0.5(낮음) / 1(중간) / 2(높음) / 3(최대) |
| **C** (Confidence) | 추정치에 대한 확신도 | 50% / 80% / 100% |
| **E** (Effort) | 구현에 필요한 인일(person-days) | 0.5 ~ 20 |

```
RICE = (Reach × Impact × Confidence) / Effort
```

### RICE → MoSCoW 자동 매핑

RICE 점수를 기준으로 MoSCoW 등급을 산출한다:

| RICE 점수 | MoSCoW | 의미 | 비율 |
|-----------|--------|------|------|
| **5.0 이상** | Must | 반드시 구현 — 없으면 서비스 불가 | ~60% |
| **2.0 ~ 4.9** | Should | 가급적 구현 — 없으면 불편하지만 가능 | ~20% |
| **0.5 ~ 1.9** | Could | 여유 있으면 구현 — 사용자 경험 향상 | ~15% |
| **0.5 미만** | Won't | 이번 릴리즈 제외 — 다음으로 연기 | ~5% |

### 구현 우선순위 테이블

spec.md의 `## 기능 요구사항` 아래에 아래 테이블을 반드시 포함한다:

```markdown
## 구현 우선순위

| FR | 기능 | R | I | C | E | RICE | MoSCoW | 구현 순서 |
|----|------|---|---|---|---|------|--------|----------|
| FR-1 | [기능명] | 8 | 2 | 100% | 3 | 5.3 | Must | 1 |
| FR-2 | [기능명] | 5 | 1 | 80% | 5 | 0.8 | Could | 4 |
```

**구현 순서 규칙:**
1. Must를 RICE 내림차순으로
2. Should를 RICE 내림차순으로
3. Could는 목록에 포함하되 "여유 시" 표기
4. Won't는 "이번 릴리즈 제외" 표기

### 비율 검증

MoSCoW 분포가 위 비율과 크게 벗어나면 재검토한다:
- Must가 80% 이상 → 스코프 축소 필요
- Won't가 30% 이상 → 기능 목록 자체 재검토 필요

## UI 속성 명시 가이드

기능 명세서에 UI 관련 속성을 명시할 때 아래 항목을 포함한다:

### 텍스트 속성
- 레이블 텍스트 (정확한 문구)
- placeholder 텍스트
- 에러 메시지 텍스트
- 도움말/힌트 텍스트
- 빈 상태 텍스트

### 상태 속성
- 활성/비활성 조건
- 표시/숨김 조건
- 읽기전용 조건
- 필수/선택 여부

### 인터랙션 속성
- 클릭/탭 시 동작
- 호버 시 동작 (데스크탑)
- 키보드 단축키 (있는 경우)
- 포커스 순서 (탭 순서)

### 데이터 속성
- 입력 제한 (최소/최대 길이, 허용 문자)
- 기본값
- 유효성 검사 규칙
- 정렬/필터 기본값

## 릴리즈 계획 구조

```markdown
## Phase 1 — MVP (목표: YYYY-MM-DD)
- [ ] Must 기능 전체
- [ ] 핵심 에러 처리
- 범위: [핵심 유저 플로우만]

## Phase 2 — 개선 (목표: YYYY-MM-DD)
- [ ] Should 기능
- [ ] UX 개선
- 범위: [부가 기능 + 사용성 향상]

## Phase 3 — 확장 (목표: YYYY-MM-DD)
- [ ] Could 기능
- [ ] 성능 최적화
- 범위: [편의 기능 + 최적화]
```
