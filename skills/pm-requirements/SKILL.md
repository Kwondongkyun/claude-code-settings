# 기능 명세 & 요구사항

## PRD (Product Requirements Document) 템플릿

모든 기능 기획은 아래 구조를 따른다. 규모에 따라 섹션을 생략할 수 있으나, 필수 항목은 반드시 포함한다.

### PRD 구조

```markdown
# [기능명] PRD

## 1. 개요 (필수)
- **목적**: 이 기능이 해결하는 문제
- **배경**: 왜 지금 이 기능이 필요한가
- **대상 사용자**: 누가 사용하는가
- **성공 지표**: 어떻게 성공을 측정하는가

## 2. 기능 요구사항 (필수)
### 2.1 핵심 기능
- FR-001: [기능 설명]
  - 수용 기준: Given/When/Then
- FR-002: [기능 설명]
  - 수용 기준: Given/When/Then

### 2.2 부가 기능
- FR-101: [기능 설명]

## 3. 비기능 요구사항
- 성능: 응답 시간 N초 이내
- 보안: 인증/인가 요구사항
- 접근성: WCAG 2.2 AA 준수
- 호환성: 지원 브라우저/디바이스

## 4. 화면 목록
| # | 화면명 | 설명 | 진입 경로 |
|---|--------|------|----------|
| 1 | ... | ... | ... |

## 5. 상태 정의 (필수)
각 화면별:
- 기본 상태 (Default)
- 로딩 상태 (Loading)
- 빈 상태 (Empty)
- 에러 상태 (Error)
- 성공 상태 (Success)

## 6. 우선순위
MoSCoW 기준으로 분류

## 7. 릴리즈 계획
- Phase 1 (MVP): ...
- Phase 2: ...
```

## 수용 기준 작성 가이드

### Given-When-Then 포맷

모든 기능 요구사항에 수용 기준을 반드시 포함한다.

```
Given: [사전 조건 — 시스템/사용자의 현재 상태]
When: [트리거 — 사용자가 수행하는 행동]
Then: [결과 — 시스템이 보여주는 응답]
```

### 작성 규칙

1. **하나의 수용 기준 = 하나의 시나리오** (여러 시나리오를 하나에 합치지 않는다)
2. **테스트 가능해야 한다** (주관적 표현 금지: "빠르게", "적절하게" → 구체적 수치)
3. **사용자 관점으로 작성** (기술 구현이 아닌 사용자가 보는 결과)

```
❌ Bad: 시스템이 적절한 에러를 반환한다
✅ Good: "이메일 형식이 올바르지 않습니다" 메시지가 입력 필드 하단에 표시된다
```

### 수용 기준 체크리스트

- [ ] 정상 동작 (Happy Path)이 정의되어 있는가?
- [ ] 실패 케이스가 정의되어 있는가?
- [ ] 경계값이 정의되어 있는가? (최소/최대)
- [ ] 권한별 동작이 정의되어 있는가?
- [ ] 응답 시간/성능 기준이 있는가?

## 우선순위: MoSCoW

기능을 4단계로 분류한다:

| 등급 | 의미 | 기준 | 비율 |
|------|------|------|------|
| **Must** | 반드시 필요 | 없으면 서비스 불가 | ~60% |
| **Should** | 있어야 함 | 없으면 불편하지만 가능 | ~20% |
| **Could** | 있으면 좋음 | 사용자 경험 향상 | ~15% |
| **Won't** | 이번엔 안 함 | 다음 릴리즈로 연기 | ~5% |

### 우선순위 판단 기준

```
사용자 가치 (High/Low) × 기술 복잡도 (High/Low)

| | 기술 복잡도 낮음 | 기술 복잡도 높음 |
|---|---|---|
| 사용자 가치 높음 | Must - 즉시 구현 | Should - 계획 후 구현 |
| 사용자 가치 낮음 | Could - 여유 있을 때 | Won't - 다음 릴리즈 |
```

## UI 속성 명시 가이드

기능 명세서에 UI 관련 속성을 명시할 때 아래 항목을 포함한다:

### 텍스트 속성
- 레이블 텍스트 (정확한 문구)
- placeholder 텍스트
- 에러 메시지 텍스트
- 도움말/힌트 텍스트
- 빈 상태 텍스트

### 상태 속성
- 활성/비활성 조건
- 표시/숨김 조건
- 읽기전용 조건
- 필수/선택 여부

### 인터랙션 속성
- 클릭/탭 시 동작
- 호버 시 동작 (데스크탑)
- 키보드 단축키 (있는 경우)
- 포커스 순서 (탭 순서)

### 데이터 속성
- 입력 제한 (최소/최대 길이, 허용 문자)
- 기본값
- 유효성 검사 규칙
- 정렬/필터 기본값

## 릴리즈 계획 구조

```markdown
## Phase 1 — MVP (목표: YYYY-MM-DD)
- [ ] Must 기능 전체
- [ ] 핵심 에러 처리
- 범위: [핵심 유저 플로우만]

## Phase 2 — 개선 (목표: YYYY-MM-DD)
- [ ] Should 기능
- [ ] UX 개선
- 범위: [부가 기능 + 사용성 향상]

## Phase 3 — 확장 (목표: YYYY-MM-DD)
- [ ] Could 기능
- [ ] 성능 최적화
- 범위: [편의 기능 + 최적화]
```
