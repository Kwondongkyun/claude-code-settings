# 사용자 스토리 & 유저 플로우

## 사용자 스토리 작성

### 표준 포맷

```
As a [사용자 유형],
I want [기능/행동],
So that [이점/목적].
```

### 수용 기준 (Acceptance Criteria)

각 스토리에 반드시 Given-When-Then 형식의 수용 기준을 포함한다.

```
Given: [사전 조건]
When: [사용자 행동]
Then: [기대 결과]
```

예시:
```
Story: 사용자가 비밀번호를 재설정할 수 있다.

Given: 사용자가 로그인 페이지에 있다
When: "비밀번호 찾기" 링크를 클릭한다
Then: 이메일 입력 폼이 표시된다

Given: 사용자가 등록된 이메일을 입력했다
When: "전송" 버튼을 클릭한다
Then: 비밀번호 재설정 이메일이 발송된다
Then: "이메일을 확인해주세요" 메시지가 표시된다

Given: 사용자가 등록되지 않은 이메일을 입력했다
When: "전송" 버튼을 클릭한다
Then: "등록되지 않은 이메일입니다" 에러 메시지가 표시된다
```

### INVEST 원칙

좋은 사용자 스토리는 INVEST 원칙을 따른다:

| 원칙 | 설명 | 체크 |
|------|------|------|
| **I**ndependent | 다른 스토리에 의존하지 않음 | 단독 개발/배포 가능한가? |
| **N**egotiable | 세부사항은 협의 가능 | 구현 방식이 고정되어 있지 않은가? |
| **V**aluable | 사용자에게 가치 제공 | 사용자가 이 기능을 왜 원하는가 설명 가능한가? |
| **E**stimable | 개발 규모 추정 가능 | 개발자가 일정을 산정할 수 있는가? |
| **S**mall | 하나의 스프린트에 완료 가능 | 너무 크지 않은가? |
| **T**estable | 테스트 가능한 수용 기준 있음 | 완료 조건이 명확한가? |

## 유저 플로우 매핑

### 플로우 유형

모든 기능에 대해 3가지 플로우를 정의한다:

1. **Primary Flow (주요 흐름)** — 가장 일반적인 성공 경로
2. **Alternative Flow (대안 흐름)** — 유효하지만 덜 일반적인 경로
3. **Error Flow (에러 흐름)** — 실패/에러 발생 시 경로

### 플로우 작성 규칙

각 단계를 다음 형식으로 정의한다:

```
[진입점] → [행동 1] → [의사결정 포인트] → [행동 2] → [완료 지점]
                              ↓
                    [대안/에러 경로]
```

### 플로우 작성 체크리스트

- [ ] 진입점이 명확한가? (어디에서 이 기능에 접근하는가?)
- [ ] 각 단계에서 사용자가 보는 화면/UI가 정의되어 있는가?
- [ ] 의사결정 포인트가 표시되어 있는가? (분기 조건)
- [ ] 에러 발생 시 복구 경로가 있는가?
- [ ] 완료 후 어디로 이동하는가?
- [ ] 뒤로가기/취소 시 어디로 돌아가는가?

### 플로우 템플릿

```markdown
## [기능명] 유저 플로우

### Primary Flow
1. 사용자가 [진입점]에서 [행동]
2. 시스템이 [응답]
3. 사용자가 [행동]
4. 시스템이 [결과] → [완료 지점]으로 이동

### Alternative Flow
- 2a. [조건]인 경우 → [대안 동작]
- 3a. [조건]인 경우 → [대안 동작]

### Error Flow
- 2e. [에러 조건] → [에러 메시지] → [복구 방법]
- 3e. [에러 조건] → [에러 메시지] → [복구 방법]
```

## 시나리오 & 엣지케이스

### 시나리오 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| Happy Path | 모든 것이 정상 작동 | 사용자가 올바른 정보로 회원가입 완료 |
| Sad Path | 사용자 실수 또는 잘못된 입력 | 이미 등록된 이메일로 가입 시도 |
| Edge Case | 예외적이지만 가능한 상황 | 이메일 주소가 255자인 경우 |

### 반드시 고려할 엣지케이스

기능 기획 시 아래 항목을 체크리스트로 사용한다:

**데이터 관련**
- [ ] 데이터가 없는 경우 (빈 상태)
- [ ] 데이터가 극단적으로 많은 경우 (1만 건 이상)
- [ ] 데이터가 극단적으로 긴 텍스트인 경우
- [ ] null/undefined 값이 들어오는 경우
- [ ] 특수문자, 이모지, 다국어 입력

**네트워크/시스템 관련**
- [ ] 네트워크 끊김/느린 연결
- [ ] API 응답 지연 (타임아웃)
- [ ] 서버 에러 (500)
- [ ] 동시 요청 (중복 클릭, 동시 편집)

**사용자 행동 관련**
- [ ] 브라우저 뒤로가기/앞으로가기
- [ ] 새로고침 (F5)
- [ ] 탭 전환 후 복귀
- [ ] 로그인 세션 만료 중 작업
- [ ] 권한이 없는 페이지 직접 URL 접근

**디바이스 관련**
- [ ] 모바일 vs 데스크탑 차이
- [ ] 다양한 브라우저 (Chrome, Safari, Firefox)
- [ ] 키보드만으로 조작 (접근성)

### 시나리오 작성 템플릿

```markdown
## [기능명] 시나리오

### Happy Path
| # | 사용자 행동 | 시스템 응답 | 비고 |
|---|-----------|-----------|------|
| 1 | ... | ... | |
| 2 | ... | ... | |

### Sad Path
| # | 상황 | 사용자 행동 | 시스템 응답 |
|---|------|-----------|-----------|
| 1 | ... | ... | ... |

### Edge Cases
| # | 엣지케이스 | 예상 동작 | 우선순위 |
|---|----------|----------|---------|
| 1 | ... | ... | Must/Should/Could |
```
