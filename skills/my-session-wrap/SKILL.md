# My Session Wrap

코딩 세션이 끝날 때 작업을 정리하고 문서화하는 스킬.
Multi-agent 2-Phase Pipeline으로 빠르고 체계적으로 정리한다.

## Execution Flow

```
┌─────────────────────────────────────────────────────┐
│  Step 1: Git 상태 확인                                │
│  - git diff, git status로 변경 내용 파악               │
│  - 변경된 파일 목록과 요약 생성                         │
├─────────────────────────────────────────────────────┤
│  Step 2: Phase 1 — 4개 분석 에이전트 (병렬 실행)       │
│  ┌──────────────┬──────────────┐                     │
│  │ doc-updater   │ automation-  │                     │
│  │ 문서 업데이트  │ scout        │                     │
│  │ 필요한 곳 탐색 │ 자동화 제안   │                     │
│  ├──────────────┼──────────────┤                     │
│  │ learning-    │ followup-    │                     │
│  │ extractor    │ suggester    │                     │
│  │ 배운 것 정리  │ 다음 할 일    │                     │
│  └──────────────┴──────────────┘                     │
├─────────────────────────────────────────────────────┤
│  Step 3: Phase 2 — 검증 에이전트 (순차 실행)           │
│  ┌─────────────────────────────────────┐             │
│  │ duplicate-checker                    │             │
│  │ Phase 1 결과에서 중복 확인 및 병합     │             │
│  └─────────────────────────────────────┘             │
├─────────────────────────────────────────────────────┤
│  Step 4: 결과 통합                                    │
│  - Phase 1 + Phase 2 결과를 하나의 리포트로 합침        │
├─────────────────────────────────────────────────────┤
│  Step 5: 사용자 선택 + 실행                            │
│  - 사용자에게 실행할 작업 선택받기                       │
│  - 선택된 작업만 실행                                  │
└─────────────────────────────────────────────────────┘
```

## Step 1: 변경 내용 확인

세션에서 어떤 작업을 했는지 자동으로 파악한다.

### 실행 방법

```bash
git diff --stat
git status --short
git log --oneline -10
```

### 수집할 정보

- 변경된 파일 목록 (신규, 수정, 삭제)
- 각 파일의 변경 요약 (추가/삭제 라인 수)
- 최근 커밋 메시지 (아직 커밋 안 한 변경사항 포함)
- 현재 브랜치명

### 출력 형식

```
변경 요약:
  - 신규 파일: 3개
  - 수정 파일: 5개
  - 삭제 파일: 1개
  - 미커밋 변경사항: 있음/없음
```

> 이 정보가 Phase 1의 4개 에이전트에게 전달된다.

## Step 2: Phase 1 — 병렬 분석 (4개 에이전트)

4개의 전문가 에이전트가 **동시에** 각자의 관점에서 세션을 분석한다.
서로 의존하지 않으므로 병렬 실행이 가능하다.

### 에이전트 1: doc-updater (문서 업데이트 탐색)

**역할**: 변경 내용을 바탕으로, 업데이트가 필요한 문서를 찾아낸다.

분석 대상:
- CLAUDE.md — 새로운 규칙이나 패턴이 추가되었는지
- README.md — 프로젝트 설명이 변경되었는지
- 기타 문서 파일 — 변경사항과 관련된 문서가 있는지

출력:
```
[doc-updater 결과]
- CLAUDE.md: "새 스킬 my-session-wrap 추가" 반영 필요
- README.md: 변경 불필요
```

### 에이전트 2: automation-scout (자동화 발견)

**역할**: 세션에서 반복된 패턴을 발견하고, 자동화를 제안한다.

분석 대상:
- 같은 명령어를 여러 번 실행했는지
- 비슷한 구조의 파일을 반복 생성했는지
- 수동으로 한 작업 중 스크립트/스킬로 만들 수 있는 것

출력:
```
[automation-scout 결과]
- "테스트 실행 후 린트 실행" 패턴 발견 → 스크립트로 자동화 가능
- 반복 패턴 없음 (해당 없음일 수도 있음)
```

### 에이전트 3: learning-extractor (학습 내용 추출)

**역할**: 오늘 세션에서 배운 것을 정리한다.

분석 대상:
- 새로 사용한 라이브러리나 API
- 해결한 에러와 해결 방법
- 새로 알게 된 패턴이나 기법

출력:
```
[learning-extractor 결과]
- multi-agent 패턴으로 병렬 분석하는 방법을 배움
- 2-Phase Pipeline으로 결과 품질을 높이는 방법을 배움
```

### 에이전트 4: followup-suggester (후속 작업 제안)

**역할**: 다음 세션에서 이어서 할 일을 제안한다.

분석 대상:
- TODO 주석이 남아있는지
- 미완성 기능이 있는지
- 테스트가 빠진 부분이 있는지
- 리팩토링이 필요한 부분이 있는지

출력:
```
[followup-suggester 결과]
- TODO: API 에러 핸들링 추가 (src/api.ts:42)
- 테스트 미작성: my-session-wrap 스킬 실행 테스트
```

## Step 3: Phase 2 — 검증 (duplicate-checker)

Phase 1의 4개 에이전트 결과를 받아서 중복을 확인한다.

### duplicate-checker 에이전트

**역할**: Phase 1 결과에서 겹치는 내용을 걸러낸다.

검증 규칙:
1. **완전 중복**: 같은 제안이 2개 이상 있으면 하나만 남기고 제거
   - 예: doc-updater "README 업데이트" + followup-suggester "README 수정" → 하나로 통합
2. **부분 중복**: 비슷하지만 관점이 다르면 병합을 제안
   - 예: doc-updater "CLAUDE.md에 스킬 추가" + learning-extractor "새 스킬 패턴 학습" → 별도 유지
3. **충돌**: 서로 모순되는 제안이 있으면 플래그
   - 예: 한쪽은 "파일 삭제 제안", 다른 쪽은 "파일 업데이트 제안" → 사용자에게 선택 요청

출력:
```
[duplicate-checker 결과]
- 중복 제거: 1건 (doc-updater + followup-suggester의 README 관련 제안 통합)
- 부분 중복: 0건
- 충돌: 0건
- 최종 제안: N건
```

## Step 4: 결과 통합 + 사용자 선택

Phase 1 + Phase 2 결과를 하나의 리포트로 합치고, 사용자에게 실행할 작업을 선택받는다.

### 통합 리포트 형식

```
═══════════════════════════════════
  Session Wrap 리포트
═══════════════════════════════════

📊 변경 요약
  - 신규 3개 / 수정 5개 / 삭제 1개

📝 문서 업데이트 제안 (doc-updater)
  1. CLAUDE.md에 my-session-wrap 스킬 추가 기록

🔄 자동화 제안 (automation-scout)
  1. "테스트+린트" 반복 패턴 → 스크립트 자동화

📚 오늘 배운 것 (learning-extractor)
  1. multi-agent 2-Phase Pipeline 패턴

📋 다음 할 일 (followup-suggester)
  1. API 에러 핸들링 추가
  2. 스킬 실행 테스트 작성

⚠️ 검증 결과 (duplicate-checker)
  - 중복 제거 1건

═══════════════════════════════════
```

### 사용자 선택

AskUserQuestion으로 실행할 작업을 선택받는다:

```
어떤 작업을 실행할까요? (복수 선택 가능)
  □ 커밋 생성 — 변경사항을 커밋합니다
  □ 문서 업데이트 — 제안된 문서 변경을 실행합니다
  □ 자동화 생성 — 제안된 스크립트/스킬을 만듭니다
  □ 건너뛰기 — 리포트만 확인하고 종료합니다
```

### 실행

사용자가 선택한 작업만 실행한다:
- **커밋 생성**: 변경사항을 스테이징하고 의미 있는 커밋 메시지로 커밋
- **문서 업데이트**: doc-updater가 제안한 문서 변경을 적용
- **자동화 생성**: automation-scout가 제안한 스크립트/스킬을 생성
- **건너뛰기**: 아무 작업도 하지 않고 리포트만 남김

## Quick Reference

### 언제 사용하면 좋은가

- 긴 코딩 세션(30분 이상)을 마칠 때
- 프로젝트를 전환하기 전에
- 하루 작업을 마무리할 때
- 중요한 기능 구현을 완료했을 때

### 언제 건너뛰어도 되는가

- 짧은 질문만 한 경우 (5분 미만)
- 코드를 읽기만 하고 변경하지 않은 경우
- 이미 수동으로 커밋과 정리를 완료한 경우
