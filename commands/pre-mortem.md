## Spec File Discovery

1. `$ARGUMENTS`가 **폴더 경로**이면 해당 폴더 내 `spec.md`를 읽는다.
2. `$ARGUMENTS`가 **파일 경로**이면 해당 파일을 읽는다.
3. `$ARGUMENTS`가 없으면 `docs/specs/` 디렉토리를 자동 탐색한다:
   - `docs/specs/` 하위의 **기능 폴더 목록**을 검색한다.
   - AskUserQuestion으로 분석할 **폴더**를 선택받는다.
   - 선택된 폴더 내 `spec.md`를 읽는다.
4. `docs/specs/` 디렉토리도 없고 `$ARGUMENTS`도 없으면, 경로를 직접 입력받는다.

spec.md를 읽은 후 아래 워크플로우를 진행한다.

## 리스크 분류 프레임워크

spec.md의 모든 FR(기능 요구사항)과 기술 구현 사항을 3가지 카테고리로 분류한다.

### Tigers (진짜 위험)
실제로 구현을 지연시키거나 실패시킬 수 있는 기술적 리스크.
- 복잡한 상태 관리 (여러 컴포넌트 간 동기화, 실시간 데이터)
- 성능 병목 (대량 데이터 렌더링, 복잡한 쿼리, 이미지/미디어 처리)
- 불명확한 외부 API/서비스 의존성 (문서 부족, rate limit, 장애 대응)
- 데이터 마이그레이션/스키마 변경 리스크
- 인증/권한 로직 복잡도
- 브라우저/디바이스 호환성 이슈

### Paper Tigers (과대평가된 위험)
어려워 보이지만 실제로는 straightforward한 구현.
- 잘 문서화된 라이브러리로 해결 가능한 기능
- 기존 코드베이스에 이미 패턴이 있는 작업
- boilerplate가 많지만 복잡하지 않은 반복 작업
- 공식 가이드/튜토리얼이 있는 통합 작업

### Elephants (방 안의 코끼리)
아무도 언급하지 않았지만 반드시 대응해야 하는 문제.
- 배포 전략 (다운타임, 롤백 계획, 환경 분리)
- 모니터링/로깅/에러 트래킹 부재
- 접근성(a11y)/국제화(i18n) 미고려
- 테스트 커버리지 부재
- SEO/OG 메타 미대응
- 데이터 정합성/백업 전략

## Workflow

### STEP 1: 리스크 식별

spec.md를 정독하고, 각 FR과 기술 요소에서 잠재 리스크를 식별한다.

**식별 관점:**
- 각 FR의 기술적 난이도와 불확실성
- FR 간 의존성/충돌
- 비기능 요구사항(성능, 보안)의 구현 난이도
- 엣지케이스 중 구현이 까다로운 항목
- 외부 서비스 의존성

### STEP 2: 인터뷰

AskUserQuestion으로 확인이 필요한 사항을 질문한다.

**Rules:**
- 질문은 구체적이고 spec에 특화되어야 한다 — 일반론 금지.
- Q1, Q2, Q3... 번호를 매긴다.
- 이전 답변을 반영하여 후속 질문을 구성한다.
- 기술 스택, 팀 경험, 인프라 환경에 대해 물어본다.
- 5~8개 질문 이내로 충분히 파악한다.

**질문 예시:**
- "FR-3의 실시간 알림은 WebSocket vs SSE 중 어떤 방식을 고려하고 있나요?"
- "현재 프로젝트에서 Three.js 등 3D 렌더링 경험이 있나요?"
- "배포는 Vercel인가요? 서버리스 제한사항은 확인했나요?"

### STEP 3: 리스크 분류 및 출력

인터뷰 결과를 반영하여 최종 분류를 작성한다.

**출력 형식:**

```markdown
# Part 4. 개발 리스크 분석 (Pre-mortem)

## Tigers (진짜 위험)

| # | FR | 리스크 | 영향도 | 대응 전략 |
|---|-----|--------|-------|----------|
| T-1 | FR-1 | [구체적 리스크] | 높음/중간 | [구체적 대응 — Phase/Task 명시] |
| T-2 | FR-3 | [구체적 리스크] | 높음 | [구체적 대응] |

## Paper Tigers (과대평가된 위험)

| # | FR | 우려 사항 | 실제 난이도 | 근거 |
|---|-----|----------|-----------|------|
| P-1 | FR-2 | [우려 사항] | 낮음 | [왜 쉬운지 근거] |

## Elephants (방 안의 코끼리)

| # | 영역 | 미대응 이슈 | 영향도 | 권장 대응 시점 |
|---|------|-----------|-------|-------------|
| E-1 | 배포 | [이슈] | 높음 | Phase 5 |
| E-2 | 테스트 | [이슈] | 중간 | Phase 7 |
```

**분류 원칙:**
- Tiger가 0개이면 의심한다 — 리스크 없는 프로젝트는 없다.
- 확신이 없으면 Tiger로 분류한다 — 과소평가보다 과대평가가 안전하다.
- 대응 전략은 Phase 번호(Phase 5 Foundation, Phase 7 검증 등)를 명시한다.

### STEP 4: spec.md 업데이트

분석 결과를 spec.md 끝에 추가한다:
- `## 미결 사항` 바로 위에 `---` 구분선과 함께 `# Part 4. 개발 리스크 분석 (Pre-mortem)` 삽입
- 기존 Part 1~3 내용은 절대 수정하지 않는다
- Tiger의 대응 전략이 기존 FR에 영향을 주는 경우, `## 미결 사항`에 `> [!WARNING]`으로 추가

완료 후 변경 요약을 출력한다.
